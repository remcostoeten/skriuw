# API Architecture Improvement Plan

> **Project:** Tauri Local-First Monorepo
> **Date:** October 2025
> **Status:** Planning Phase

---

## Executive Summary

This document outlines a comprehensive plan to improve the API layer and data fetching patterns across both InstantDB and Turso implementations. The goal is to achieve consistency, performance, scalability, and excellent developer experience while maintaining the local-first architecture.

### Key Findings
- ✅ **InstantDB** has solid foundation with core hook abstractions
- ❌ **Turso** lacks abstraction layer and has significant boilerplate
- ⚠️ Both need performance optimizations (memoization, request deduplication)
- ⚠️ No standardized error handling or cache management strategy

---

## Current State Analysis

### InstantDB Strengths
- ✅ Reusable core hooks (`useCreate`, `useUpdate`, `useRead`, `useDestroy`)
- ✅ Domain hooks are concise (15-25 lines)
- ✅ Automatic optimistic updates
- ✅ Consistent error/loading states
- ✅ Clear separation of concerns

### InstantDB Weaknesses
1. **No memoization** - handlers recreated every render
2. **Weak type safety** - `entityName: string` not validated against schema
3. **No error recovery** - no retry logic, no global error boundaries
4. **No request deduplication**

### Turso Critical Issues
1. **No core abstraction** - every hook repeats 15-20 lines of boilerplate
2. **Manual cache invalidation** - easy to forget `refetch()`, causes stale data bugs
3. **No optimistic updates** - slower perceived performance
4. **Refetch prop drilling** - `onUpdate={refetch}` passed through components

---

## Strategic Decision: Custom Hooks vs React Query

### Option A: Improve Custom Hooks ⭐ FOR LEARNING

**Pros:**
- Full control, no dependencies
- Educational value
- Lightweight (~2KB)
- Tailored to exact needs

**Cons:**
- More dev time
- Maintenance burden
- Need to implement features manually

### Option B: Migrate to TanStack Query ⭐ FOR PRODUCTION

**Pros:**
- All features built-in (cache, deduplication, optimistic updates, retry, devtools)
- Battle-tested (10M+ weekly downloads)
- Excellent docs + community
- 12KB gzipped
- Framework agnostic

**Cons:**
- Learning curve
- Additional dependency
- Less control over internals

**Recommendation:** Start with custom hooks to understand patterns deeply, then migrate to React Query for production scaling.

```typescript
// apps/turso/src/hooks/core/use-mutation.ts
import { useState, useCallback } from 'react';
import { cacheManager } from '@/lib/cache-manager';

interface UseMutationOptions {
  onSuccess?: (data: TData) => void | Promise;
  onError?: (error: Error) => void;
  invalidates?: string[]; // Cache keys to invalidate
}

export function useMutation(
  mutationFn: (variables: TVariables) => Promise,
  options?: UseMutationOptions
) {
  const [isLoading, setIsLoading] = useState(false);
  const [error, setError] = useState(null);

  const mutate = useCallback(
    async (variables: TVariables) => {
      try {
        setIsLoading(true);
        setError(null);

        const data = await mutationFn(variables);

        // Automatically invalidate caches
        if (options?.invalidates) {
          options.invalidates.forEach(key => {
            cacheManager.invalidate(key);
          });
        }

        await options?.onSuccess?.(data);
        return data;
      } catch (err) {
        const error = err as Error;
        setError(error);
        options?.onError?.(error);
        throw error;
      } finally {
        setIsLoading(false);
      }
    },
    [mutationFn, options]
  );

  return { mutate, isLoading, error };
}
```

```typescript
// apps/turso/src/hooks/core/use-query.ts
import { useState, useEffect, useCallback } from 'react';
import { cacheManager } from '@/lib/cache-manager';

interface UseQueryOptions {
  enabled?: boolean;
  cacheKey: string;
  onSuccess?: (data: TData) => void;
  onError?: (error: Error) => void;
}

export function useQuery(
  queryFn: () => Promise,
  options: UseQueryOptions
) {
  const [data, setData] = useState(undefined);
  const [isLoading, setIsLoading] = useState(true);
  const [error, setError] = useState(null);

  const refetch = useCallback(async () => {
    try {
      setIsLoading(true);
      const result = await queryFn();
      setData(result);
      setError(null);
      options.onSuccess?.(result);
    } catch (err) {
      const error = err as Error;
      setError(error);
      options.onError?.(error);
    } finally {
      setIsLoading(false);
    }
  }, [queryFn, options]);

  useEffect(() => {
    if (options.enabled === false) return;

    refetch();

    // Subscribe to cache invalidation
    cacheManager.subscribe(options.cacheKey, refetch);

    return () => {
      cacheManager.unsubscribe(options.cacheKey, refetch);
    };
  }, [options.cacheKey, options.enabled, refetch]);

  return { data, isLoading, error, refetch };
}
```

#### 1.2 Implement Cache Manager

**File to create:**
```
apps/turso/src/lib/cache-manager.ts
```

```typescript
type CacheKey = string;
type InvalidationFn = () => void;

class CacheManager {
  private listeners = new Map>();

  subscribe(key: CacheKey, callback: InvalidationFn) {
    if (!this.listeners.has(key)) {
      this.listeners.set(key, new Set());
    }
    this.listeners.get(key)!.add(callback);
  }

  unsubscribe(key: CacheKey, callback: InvalidationFn) {
    this.listeners.get(key)?.delete(callback);
  }

  invalidate(key: CacheKey) {
    const callbacks = this.listeners.get(key);
    if (callbacks) {
      callbacks.forEach(cb => cb());
    }
  }

  invalidatePattern(pattern: RegExp) {
    Array.from(this.listeners.keys())
      .filter(key => pattern.test(key))
      .forEach(key => this.invalidate(key));
  }
}

export const cacheManager = new CacheManager();
```

#### 2.2 Request Deduplication

**Problem:** Multiple components calling the same query → duplicate requests

**Solution:**
```typescript
// apps/turso/src/lib/request-cache.ts
const inflightRequests = new Map>();

export async function dedupedRequest(
  key: string,
  fn: () => Promise
): Promise {
  if (inflightRequests.has(key)) {
    return inflightRequests.get(key)!;
  }

  const promise = fn().finally(() => {
    inflightRequests.delete(key);
  });

  inflightRequests.set(key, promise);
  return promise;
}
```

**Integrate into `useQuery`:**
```typescript
const refetch = useCallback(async () => {
  try {
    setIsLoading(true);

    setData(result);
  } catch (err) {
    setError(err as Error);
  } finally {
    setIsLoading(false);
  }
}, [queryFn, options.cacheKey]);
```

#### 2.3 Optimistic Updates (Turso)

**Goal:** Match InstantDB's instant UI updates

**Implementation:**
```typescript
// apps/turso/src/hooks/core/use-optimistic-mutation.ts
export function useOptimisticMutation(
  mutationFn: (variables: TVariables) => Promise,
  options: {
    cacheKey: string;
    updater: (oldData: any, variables: TVariables) => any;
    invalidates?: string[];
  }
) {
  const { mutate, isLoading, error } = useMutation(mutationFn, {
    invalidates: options.invalidates,
  });

  const optimisticMutate = useCallback(
    async (variables: TVariables) => {
      // 1. Update cache optimistically
      const rollback = cacheManager.updateOptimistically(
        options.cacheKey,
        (old) => options.updater(old, variables)
      );

      try {
        // 2. Perform actual mutation
        const result = await mutate(variables);
        return result;
      } catch (error) {
        // 3. Rollback on error
        rollback();
        throw error;
      }
    },
    [mutate, options]
  );

  return { mutate: optimisticMutate, isLoading, error };
}
```

**Usage example:**
```typescript
export function useUpdateTask() {
  return useOptimisticMutation(
    async ({ id, data }: { id: string; data: Partial }) => {
      await db.update(tasks).set(data).where(eq(tasks.id, id));
    },
    {
      cacheKey: 'tasks',
      updater: (oldTasks: Task[], { id, data }) =>
        oldTasks.map(task => task.id === id ? { ...task, ...data } : task),
      invalidates: ['tasks'],
    }
  );
}
```

#### 4.2 Add Retry Logic to Mutations

**Implementation:**
```typescript
interface UseMutationOptions {
  // ... existing options
  retry?: number | ((error: Error, attempt: number) => boolean);
  retryDelay?: number | ((attempt: number) => number);
}

async function retryWithBackoff(
  fn: () => Promise,
  options: { retry: number; retryDelay: (attempt: number) => number }
): Promise {
  let lastError: Error;

  for (let attempt = 0; attempt <= options.retry; attempt++) {
    try {
      return await fn();
    } catch (error) {
      lastError = error as Error;

      if (attempt < options.retry) {
        const delay = options.retryDelay(attempt);
        await new Promise(resolve => setTimeout(resolve, delay));
      }
    }
  }

  throw lastError!;
}
```

**Usage:**
```typescript
useMutation(fn, {
  retry: 3,
  retryDelay: (attempt) => Math.min(1000 * 2 ** attempt, 10000), // Exponential backoff
})
```

#### 4.3 User-Friendly Error Messages

**File:** `lib/error-messages.ts`

```typescript
export function getErrorMessage(error: Error): string {
  const message = error.message.toLowerCase();

  if (message.includes('sqlite_busy')) {
    return 'The database is currently busy. Please try again.';
  }

  if (message.includes('unique constraint')) {
    return 'This item already exists.';
  }

  if (message.includes('foreign key constraint')) {
    return 'Cannot delete this item because it is being used.';
  }

  if (message.includes('network') || message.includes('fetch')) {
    return 'Network error. Please check your connection.';
  }

  if (message.includes('timeout')) {
    return 'Request timed out. Please try again.';
  }

  return 'An unexpected error occurred. Please try again.';
}
```

## Final Thoughts

This plan transforms your API layer from a collection of ad-hoc patterns into a cohesive, scalable architecture. The custom hooks approach gives you full control and deep understanding, while the optional React Query migration path provides an exit strategy if you need more features or better performance.

**Key Principle:** Build abstractions that eliminate repetitive code while maintaining clarity and type safety. Every hook should be a thin wrapper around your core patterns, making it easy to understand what's happening and debug when needed.

**Start small:** Implement Phase 1 first, validate it works, then proceed. Don't try to do everything at once.
