# Architecture

Technical overview of Servo's design and implementation.

## Technology Stack

### Core Technologies

- **Go 1.21+** - Primary language
- **Bubble Tea** - Terminal UI framework
- **Lipgloss** - Terminal styling library


## Architecture Patterns

### State Machine

Servo uses a state machine pattern with Bubble Tea:

```go
type appState int

const (
    stateMenu appState = iota
    stateRunning
    stateBuilding
    stateDeploying
)
```

**State Transitions:**
- `stateMenu` → `stateRunning` (start server)
- `stateMenu` → `stateBuilding` (start build)
- `stateMenu` → `stateDeploying` (start deploy)
- Any state → `stateMenu` (return/cancel)

### Model-Update-View

Bubble Tea's MVU pattern:

**Model:**
```go
type model struct {
    config        *ServoConfig
    state         appState
    menuStack     []*MenuContext
    currentMenu   *MenuContext
    serverProcess *ServerProcess
    buildProcess  *BuildProcess
    // ...
}
```

**Update:**
```go
func (m model) Update(msg tea.Msg) (tea.Model, tea.Cmd) {
    switch m.state {
    case stateMenu:
        return m.updateMenu(msg)
    case stateRunning:
        return m.updateRunning(msg)
    // ...
    }
}
```

**View:**
```go
func (m model) View() string {
    switch m.state {
    case stateMenu:
        return m.viewMenu()
    case stateRunning:
        return m.viewRunning()
    // ...
    }
}
```

```
  ↓

  ↓

```

```
  ↓

  ↓
```

```
  - stdout scanner
  - stderr scanner
  - port detector
  ↓

  ↓
Update model with output
  ↓
Render view
```

## Key Algorithms

### Project Detection

**Monorepo Detection:**
1. Check `workspaces` field in `package.json`
2. Check for `apps/` or `packages/` directories
3. Return true if either found

**Package Manager Detection:**
1. Check `packageManager` field
2. Check lock files (priority order)
3. Default to npm

**App Discovery:**
1. Get workspace directories
2. For each directory:
   - Read `package.json`
   - Check for `dev` script
   - Extract app name
   - Check for Tauri scripts
3. Return discovered apps

### Port Detection

**Regex Patterns:**
```go
patterns := []string{
    `localhost:(\d+)`,
    `http://[^:]+:(\d+)`,
    `port (\d+)`,
    `0\.0\.0\.0:(\d+)`,
    `127\.0\.0\.1:(\d+)`,
    `:(\d{4,5})\b`,
}
```

**Detection Process:**
1. Scan each output line
2. Try each regex pattern
3. Extract port number
4. Store in process struct

### Process Killing

**Port-Based Killing:**
1. Run `lsof -ti:PORT`
2. Parse PIDs from output
3. Run `kill -9 PIDs`
4. Return success/error

**Pattern-Based Killing:**
1. Run `pkill -f PATTERN`
2. Check exit code
3. Return success/error

## Extensibility

### Adding New Menu Items

**In `internal/menu/menu.go`:**
```go
items = append(items, MenuItem{
    Name: "New Action",
    Icon: "🔧",
    Type: MenuTypeUtilityAction,
    UtilityAction: myNewFunction,
})
```

### Adding New Actions

**Define action function:**
```go
func myNewAction() (string, error) {
    // Do something
    return "Success message", nil
}
```

**Add to menu:**
```go
MenuItem{
    UtilityAction: myNewAction,
}
```

### Adding New States

**1. Add state constant:**
```go
const (
    stateMenu appState = iota
    stateRunning
    stateNewState  // Add here
)
```

**2. Add update handler:**
```go
func (m model) updateNewState(msg tea.Msg) (tea.Model, tea.Cmd) {
    // Handle messages
}
```

**3. Add view function:**
```go
func (m model) viewNewState() string {
    // Render UI
}
```

**4. Wire up in Update/View:**
```go
case stateNewState:
    return m.updateNewState(msg)
```

### Custom Styling

**In `internal/app/styles.go`:**
```go
var myNewStyle = lipgloss.NewStyle().
    Foreground(textPrimary).
    Bold(true)
```

## Performance Considerations

### Output Buffering

**Server Output:**
- Buffer size: 200 lines
- Circular buffer (drops old lines)
- Thread-safe with mutex

**Build Output:**
- Buffer size: 100 lines
- Similar circular buffer
- Efficient memory usage

### Process Management

**Goroutines:**
- Separate goroutines for stdout/stderr
- Non-blocking output scanning
- Clean shutdown on exit

**Resource Usage:**
- Low memory footprint
- Minimal CPU usage
- Efficient I/O operations

## Error Handling

### Graceful Degradation

**Configuration Errors:**
- Fallback to empty config
- Continue with limited functionality
- Show error messages

**Process Errors:**
- Capture and display errors
- Allow user to retry
- Clean up on failure

**Detection Errors:**
- Default to safe values
- Log errors (future)
- Continue operation

Servo's architecture prioritizes:
- **Simplicity** - Easy to understand and modify

The codebase is organized for maintainability and future growth.
